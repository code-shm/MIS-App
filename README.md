# Amex-Based Complaints Analytics & MIS Platform

An end-to-end **management-information & predictive-analytics system** for
consumer complaints, built to simulate American Express service-operations
functions. It ingests the public **CFPB Consumer Complaint Database**, enriches
unstructured complaint narratives with NLP sentiment, trains supervised models
to flag **escalation** and **churn risk**, benchmarks Amex against the whole
industry, and serves the results through an interactive dashboard (with a
parallel Power BI model) and a BigQuery-ready data layer.

> **Data is 100% real and public.** Sourced live from the U.S. Consumer
> Financial Protection Bureau (CFPB). No synthetic complaints. The only modelled
> construct is the churn-risk *label*, which is a transparent, documented proxy
> (see [Churn methodology](#churn-risk-methodology)).

## Live dashboard

📊 **[View the interactive dashboard](https://claude.ai/code/artifact/5faf3056-f02f-44b2-a89e-e070daef6e55)**
&nbsp;·&nbsp; or open `dashboards/html/amex_dashboard_standalone.html` locally (static snapshot, no server needed).

### Dashboard features

- **Overview** — KPI cards, monthly volume + risk trends, product/issue/state
  breakdowns, response mix, and the industry benchmark.
- **Alerts & anomalies** — auto-detected from the complaint time-series (robust
  median/MAD z-scores; trailing baseline for volume, global for rates) and risk
  segments: a "needs attention" read on the latest complete month, product/issue
  risk hotspots (escalation/churn vs. the Amex average), notable historical
  spikes, and an anomaly timeline. A tab badge shows the open critical/serious count.
- **Complaint Explorer** — searchable, sortable, filterable table of the
  highest-risk individual complaints (escalation %, churn-risk, sentiment); click
  a row to read the narrative or send it straight to the scorer.
- **Live Scorer** — paste any complaint text and get an escalation probability
  and VADER sentiment computed **entirely in the browser** from the exact trained
  models (the JS port is validated to 0.00000 deviation vs. the Python pipeline —
  see `scripts/validate_scorer.py`), including the words driving the prediction.

The dashboard is fully **responsive** (scrollable tabs, fluid grids, touch-sized
targets) and theme-aware (light/dark).

### Agentic (self-updating) mode

Run it as a live service that keeps pulling new complaints and lets you refresh
on demand:

```bash
python -m src.serve                 # serve + auto-refresh every 60 min
python -m src.serve --interval 15   # auto-refresh every 15 minutes
python -m src.serve --no-auto       # serve only; use the "Refresh now" button
```

Then open **http://localhost:8000/index.html**. The header shows a live
"● Updated …" pill and a **🔄 Refresh now** button. Behind the scenes:

- a background loop re-pulls the CFPB Amex feed on your interval,
- `src/refresh.py` scores only *new* complaints (cached VADER for known ones),
  then re-stamps `version.json`,
- the dashboard polls `/api/status` and **hot-reloads** its data when the stamp
  changes — no page refresh needed.

You can also trigger a refresh from anywhere: `curl -X POST
http://localhost:8000/api/refresh` (add `?retrain=1` to retrain the models), or
run `python -m src.refresh` as a one-off / cron job.

### PDF reports

- **In the dashboard**, the **⬇ PDF** button downloads a polished multi-page
  **executive report** (KPI summary, trends, breakdowns, model cards, industry
  benchmark) generated by the server on demand. Off-server (static file) it
  falls back to the browser's print-to-PDF, which the page is styled for.
- **From the CLI**: `npm run report` (or `python -m src.report`) writes
  `reports/amex_executive_report.pdf`.
- **Endpoint**: `GET /api/report.pdf` streams a freshly generated report.

## What it does

| Layer | Scope | Output |
|-------|-------|--------|
| **Industry MIS** | The full **17M+**-complaint national dataset (8,069 companies) | Monthly volume / timely / relief benchmarks; company league table |
| **Ingestion** | **58,169** American Express complaints (2011–2026) | Typed Parquet working set |
| **NLP enrichment** | Narratives (25,016 non-empty) | VADER sentiment scores, labels & bands |
| **Escalation model** | Amex complaints | Probability a case needs relief / breaches SLA |
| **Churn-risk model** | Amex complaints | Early-warning at-risk probability per customer |
| **Serving** | Curated tables | Interactive HTML dashboard · Power BI model · BigQuery |

## Quickstart

### Via npm (recommended)

The analytics/ML core is Python; a Node CLI (`bin/amex.js`) lets you drive
everything through npm on any OS.

```bash
npm run setup        # pip install -r requirements.txt
npm start            # launch the live self-updating dashboard (http://localhost:8000)
npm run pipeline     # full rebuild: ingest -> VADER -> models -> dashboard
npm run refresh      # pull latest complaints + rescore
npm run report       # generate the executive PDF  (reports/amex_executive_report.pdf)
npm run mis          # industry MIS over the 17M+ bulk export
npm run powerbi      # export Power BI CSVs
npm run all          # pipeline + mis + report + exports
```

Pass flags after `--`, e.g. `npm run serve -- --interval 15`.

### Via Python directly

```bash
pip install -r requirements.txt
python scripts/run_all.py         # full build
python scripts/run_all.py --fast  # skip the 17M-row industry pass
```

The 1.4 GB industry bulk file is downloaded once to `data/raw/`:

```bash
curl -L https://files.consumerfinance.gov/ccdb/complaints.csv.zip \
     -o data/raw/complaints_full.csv.zip
```

### Run individual stages

```bash
python -m src.ingest --amex          # fetch the Amex subset
python -m src.pipeline               # clean → VADER → train → score → dashboard
python -m src.pipeline --from-enriched   # retrain models only (skip re-scoring VADER)
python -m src.mis_aggregate          # industry MIS over the 17M+ bulk export
python scripts/export_powerbi.py     # Power BI CSVs
python -m src.bigquery_upload --project my-proj --dataset amex_complaints
```

## Modelling notes

### Escalation label
A complaint is **escalated** when it could not be closed with a plain
explanation — it forced tangible remediation (`Closed with monetary relief` or
`Closed with non-monetary relief`) or breached the timely-response SLA. None of
the label-defining fields are used as model features, so the reported AUC
reflects genuine prediction from the product/issue taxonomy, geography, channel,
handling time and the TF-IDF narrative — not label reconstruction. The model is
a class-weighted logistic regression, kept interpretable so the strongest
escalation-driving terms are inspectable (governance-friendly).

### Churn-risk methodology
The public dataset has **no account-closure signal**, so churn is a transparent
service-recovery *proxy* grounded in CX research: a customer is at risk when a
**negative-sentiment** complaint is compounded by a **service failure the company
controls** (untimely handling, no relief, or a dispute-style outcome). Crucially,
the model does **not** re-emit that rule — the proxy defines the *label*, and a
gradient-boosted model then learns to predict it from **intake-time features
only** (product / issue taxonomy, geography, channel, complaint size). The
outcome/sentiment columns that define the label are explicitly excluded from the
feature set (enforced by an assertion in `churn_model.py`), which is why the AUC
is a believable 0.879 rather than a leaked 1.0. The output is a calibrated
at-risk probability that prioritises retention outreach.

## Deployment

The static dashboard deploys to **Vercel** in one click (`vercel.json` points at
`dashboards/html/`); on a static host it runs in snapshot mode and stays fresh
via a scheduled **GitHub Action** that regenerates the data and pushes. For the
full live server (real-time Refresh button + on-demand PDF), deploy `src/serve.py`
to Render/Railway/Fly. Full guide: [`docs/DEPLOY.md`](docs/DEPLOY.md).
